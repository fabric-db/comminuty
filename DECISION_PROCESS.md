# Decision Process

Fabric decisions follow a visible path from idea to responsibility.

```text
Idea -> Proposal -> Discussion -> Review -> Decision -> Responsibility -> Evidence -> Trust Update
```

## 1. Idea

Anyone may open an issue or discussion with an idea.

Good ideas include the problem statement, affected people, expected impact, risks, and alternatives.

## 2. Proposal

Material changes should become proposals.

Use proposals for governance changes, architecture changes, new community programs, partnerships, role changes, process changes, and security-sensitive policy changes.

## 3. Discussion

Discussion should be open, respectful, and time-bounded when possible.

The goal is to improve the proposal and understand tradeoffs.

## 4. Review

Reviewers check clarity, community benefit, technical feasibility, governance impact, security impact, maintenance cost, and reversibility.

## 5. Decision

A decision must be recorded as accepted, rejected, deferred, or escalated.

Every decision should include context, decision, reason, evidence, owner, and follow-up date when needed.

## 6. Responsibility

Accepted work needs an owner.

No accepted proposal should be left without responsibility.

## 7. Evidence

Evidence may include issue links, pull requests, commits, meeting notes, test results, release notes, and public artifacts.

## 8. Trust update

Contribution and responsibility should update community trust records where appropriate.
